# 常见问题与解决（ops-data-multistep-query）

按"现象 → 原因 → 解决"三段式组织。本技能基于 chrome-devtools CLI 与 ManageOne AI 助手实测总结。

---

## 一、chrome-devtools 服务相关

### 1. `chrome-devtools: command not found`

- **现象**：执行任何 `chrome-devtools` 命令报命令未找到。
- **原因**：未全局安装 `chrome-devtools-mcp`，或 npm 全局 bin 目录不在 PATH。
- **解决**：`npm i chrome-devtools-mcp@latest -g`；确认 `npm bin -g` 路径在 PATH 中。

### 2. 命令长时间无响应 / daemon 未运行

- **现象**：`status` 返回未运行，或命令挂起。
- **原因**：MCP 未启动，或上次 `stop` 未干净退出残留进程占用端口。
- **解决**：
  ```bash
  chrome-devtools start --acceptInsecureCerts true --headless false
  ```
  若启动失败提示端口占用，查端口并强杀残留：
  ```bash
  netstat -ano | grep -E '9222|LISTENING'      # 找占用进程 PID
  taskkill /PID <pid> /F                        # 强杀
  ```

## 二、页面导航 / SSL 相关

### 3. `ERR_CERT_AUTHORITY_INVALID` / "您的连接不是私密连接"

- **现象**：导航后停在 chrome-error 页，提示证书不受信任。
- **原因**：站点使用自签名证书；且导航会多级重定向到 `auth.*` 域，每次跳转都可能触发新的证书错误。
- **解决**（按优先级）：
  1. **首选**：自启 MCP 时带 `--acceptInsecureCerts true`（步骤 1），一次性跳过所有层级。
  2. **共享 MCP 不能重启时**：循环点击绕过（最多 5 次）——点 `高级` → 点 `继续前往...(不安全)` → sleep 5 → 重新取快照，直到进入登录页/首页。

### 4. 导航超时 `--timeout`

- **现象**：`navigate_page` 报 timeout。
- **原因**：网络慢或重定向链长。
- **解决**：把 `--timeout` 提到 30000~60000；导航后用 `take_snapshot` 确认实际落地页（可能已到登录页）。

## 三、登录相关

### 5. 登录按钮点了没反应 / 仍在登录页

- **现象**：点 `登 录` 后取快照仍是登录页。
- **原因**：`fill` 后表单校验未完成，按钮短暂 disabled；或密码特殊字符未正确写入。
- **解决**：`fill` 后 `sleep 2`，**重新取快照拿到最新的登录按钮 uid** 再点；密码含 `#$%` 等字符时用单引号包裹 `'ManageOne12#$%'`，无需转义。仍失败则再重试一次。

### 6. 登录后停在加载/busy 状态

- **现象**：URL 已跳到首页但快照含 `busy` 或内容不全。
- **原因**：首页异步加载中。
- **解决**：`sleep 5` 后重新取快照，确认出现 `ManageOne AI助手` 与 `智能助手` 再继续。

## 四、云平台切换相关

### 7. 找不到云平台选择器 / 下拉不出现

- **现象**：点导航栏后没有候选列表。
- **原因**：点错了元素（点到图标而非容器）；或快照陈旧。
- **解决**：重新 `take_snapshot`，在 banner 区域找**含当前平台代号文本**（如 `per3W`/`region861vt`）的 generic 容器再点；下拉出现搜索框即成功。

### 8. 选错平台 / 候选项不匹配

- **现象**：下拉列表里没有目标平台名。
- **原因**：目标平台在该环境不存在，或名称含空格/大小写差异。
- **解决**：在下拉搜索框 `type_text "<平台关键字>"` 过滤后再点；确实不存在则告知用户并沿用当前平台继续。

## 五、对话框 / 提交相关

### 9. 输入框文本没提交 / 字数计数为 0

- **现象**：`type_text --submitKey Enter` 后对话框无新问题、输入框空。
- **原因**：该输入框是多行控件，`--submitKey` 不可靠；且焦点可能丢失。
- **解决**：改三步法——`click "<输入框 uid>"`（先获取焦点）→ `type_text` → 取快照确认 `value` 已写入、字数>0 → 单独 `press_key "Enter"`。

### 10. 回车变成换行而不是提交

- **现象**：回车后输入框内出现换行，问题未发出。
- **原因**：部分状态下 Enter 绑定为换行（Shift+Enter 才换行的约定反转）。
- **解决**：取快照找输入框旁的发送按钮（通常是 `button` 或 `image` 图标），改用 `click "<发送按钮 uid>"` 提交。

### 11. 对话框会话被重置（历史消失）

- **现象**：取快照发现对话框回到欢迎页/推荐问题，之前的问答记录消失。
- **原因**：页面刷新、对话框被关闭重开、或会话超时清空。
- **解决**：不要慌——重新 `take_snapshot` 确认当前状态；若输入框可用则直接继续提交下一轮；关键结论应在前几轮提取后**立即记录到本地**（上下文/文件），不要依赖页面历史。

## 六、AI 回答 / 数据检索相关（本技能核心）

### 12. `数据检索失败，请联系系统管理员处理`

- **现象**：某轮查询出现"数据检索失败"。
- **原因**：AI 首个数据集（如"服务监控指标性能数据数据集"）检索异常。
- **关键结论：不要判定为最终失败**。系统通常自动用另一个数据集（`SYS.SERVER物理服务器性能分析数据集`）重试，并在原回答下方**追加新的分析块**。
- **解决**：继续按 `pollIntervalSeconds` 轮询 `take_snapshot`，直到出现 `分析完成` 的成功结果，或累计达 `maxWaitSeconds`。单轮整体耗时约 1.5~3 分钟属正常。

### 13. `抱歉，助手暂时无法回答您的问题`

- **现象**：AI 直接拒绝回答。
- **原因**：多为瞬时失败或问题表述导致匹配不到数据集，与"数据检索失败"不同（这个不会自动重试）。
- **解决**：**重新提交一次**原问题（最多重试 1 次）；仍失败则微调措辞（如把宿主机名称换成更完整的 ID，或拆得更原子）。

### 14. 长时间停在"深度思考中/正在执行查询"

- **现象**：超过 `maxWaitSeconds` 仍未 `分析完成`。
- **原因**：后端检索卡住或数据量大。
- **解决**：先看 `list_console_messages --types error` 有无报错；累计超时则把本轮记为"超时未拿到结果"，继续下一轮或告知用户，不要无限等待。

### 15. 提取不到结构化数据（趋势/对比类问题默认用 ECharts 图表展示）

- **现象**：问趋势、多数据对比时，AI 返回的是 ECharts 图表（canvas/SVG 渲染），a11y 快照只能拿到一句文字摘要，`evaluate_script` 用 `document.querySelectorAll('table tbody tr')` 抓不到任何 `<table>`。
- **原因**：AI 助手对"趋势""对比"类问题**默认渲染 ECharts 图表**，DOM 中没有表格结构，只有图表附带的摘要文字（如周 max/min），无法直接抓取逐日/逐项原始数据。
- **解决（首选，在提问阶段解决）**：**在问题中明确指明"用表格展示"/"以表格形式返回原始数据"**，让 AI 改用表格输出。例如把
  `查询宿主机X最近一周的CPU使用率趋势`
  改写为
  `查询宿主机X最近一周的CPU使用率趋势，用表格展示`
  这样 DOM 中才会出现 `<table>`，可用 `evaluate_script` 抓 `document.querySelectorAll('table tbody tr')` 取到完整的逐日/逐项原始数据。表格如果有分页时，需要通过翻页获取到所有数据。
- **解决（兜底，已拿到图表时）**：若问题已发出且只有图表，只能读 AI 的文字摘要（如周 max/min），如实使用、不编造逐日序列；或改用"数据检索工作台"导出原始数据后重新提问。
- **建议**：本技能步骤 7 拆解出趋势/对比类子问题时，**默认在问句末尾追加"用表格展示"**，从源头确保数据可提取（见 SKILL.md 步骤 7）。

## 七、资源清理相关

### 16. `close_page <index>` 报页面不存在

- **现象**：用步骤 1 记录的索引关页面失败。
- **原因**：运行中页面索引变化（新建/关闭导致顺序变）。
- **解决**：清理前**重新 `list_pages`**，按 **URL/标题** 匹配本技能创建的页面拿到当前索引，再 `close_page`。

### 17. `stop` 后浏览器/进程仍在

- **现象**：`stop` 返回了，但还有 chrome.exe 进程。
- **原因**：`stop` 只回收本 MCP 启动的浏览器；用户日常 Chrome 不受影响（不该被关）。
- **解决**：先 `status` 确认 MCP 已停；再用命令行区分进程——只清理 user-data-dir 指向临时/`.cache/chrome-devtools-mcp` 的进程，**绝不**杀使用默认 User Data 的用户 Chrome：
  ```bash
  powershell -NoProfile -Command "Get-CimInstance Win32_Process -Filter \"name='chrome.exe'\" | Where-Object { \$_.CommandLine -match 'chrome-devtools-mcp|remote-debugging' } | ForEach-Object { taskkill /PID \$_.ProcessId /F }"
  ```
  无匹配输出 = 无残留，无需操作。

### 18. 误关了用户的 Chrome

- **现象**：清理时把用户正在用的浏览器也关了。
- **原因**：用 `taskkill /im chrome.exe /f` 一刀切。
- **解决**（预防）：严禁按进程名一刀切；只按命令行特征精准匹配本 MCP 启动的实例（见 17）。复用的已登录页面、共享 MCP 一律不清理。
