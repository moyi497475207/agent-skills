# 用户侧 Agent 对接云平台 AI 运维助手 — 业务场景设计

> **架构概述：** 用户侧 Agent 通过 `chrome-devtools-cli` 驱动浏览器，模拟用户操作访问云平台 AI 助手页面，在对话框中输入问题、获取回答，再将结果提取并展示到用户侧 Agent 页面。
>
> 用户全程在用户侧 Agent 界面完成交互，无需跳转到云平台。

---

## 一、交互架构

```
用户 ←→ 用户侧 Agent 页面 ←→ Agent 后端
                                  │
                          chrome-devtools-cli
                                  │
                          云平台 AI 助手页面（浏览器渲染）
                                  │
                          云平台后端（知识问答/故障诊断/数据检索）
```

**核心流程：**

1. 用户在用户侧 Agent 页面发起自然语言请求
2. Agent 后端将复合请求**拆解为原子子问题**（AI 助手一次只接受一个原子问题，复合/多条件问题无法直接回答）
3. 通过 `chrome-devtools-cli` 访问云平台 AI 助手页面（含 SSL 自签名证书处理、登录、切换到目标云平台）
4. **每一步操作前重新 `take_snapshot` 动态定位元素**，绝不硬编码 selector/UID（页面 DOM 随版本变化）
5. 在对话框中**串行**逐个提交子问题（多行输入框用「点击 → 输入 → 单独回车」三步，`submitKey` 在该控件上不可靠）
6. **轮询**等待回答：数据检索类问题单轮耗时 **30s ~ 3min**，需按阶段状态（分析中 → 筛选数据集 → 生成查询 → 执行 → 分析完成）判断完成
7. 提取回答文本/表格（趋势/对比类问题需在提问时追加「用表格展示」，否则 AI 渲染为 ECharts 图表、无法抓取原始数据）
8. Agent 后端对原始回答做二次加工（格式化、去重关联、补充上下文、合并多轮结果），**如实呈现数据粒度**（AI 返回周 max/min 就用该粒度，不编造逐日序列）
9. 将最终结果展示在用户侧 Agent 页面
10. 资源清理收尾（仅清理本次自启的浏览器/MCP，复用的登录态与会话严禁回收）

## 二、业务场景细化

### 场景 1：信息查询

**用户在 Agent 页面输入：**
> "订单服务的节点现在什么状态？"

**Agent 后台流程：**
1. 通过 chrome-devtools-cli 打开云平台 AI 助手页面（含 SSL 自签名证书处理、登录、切换到目标云平台）
2. 在对话框输入原子问题："查询订单服务所有节点的实时状态"
3. 轮询等待 AI 助手返回结果（数据检索类单轮耗时 **约 30s ~ 3min**）
4. 提取回答内容（节点列表、状态、版本等）
5. **二次加工：** 格式化为结构化卡片展示，补充节点角色标注

> ⏱ **耗时预期：** 单条信息查询是最轻量场景。若 Agent 已保持登录态，端到端 ≈ 单轮查询 30s~3min；冷启动（首次登录 + 切换平台）额外 +30~60s。

**Agent 页面展示：**
```
📦 订单服务 - 节点状态
┌─────────────┬────────┬─────────┬──────────┐
│ 节点         │ 状态   │ CPU     │ 版本     │
├─────────────┼────────┼─────────┼──────────┤
│ orderer-01  │ ✅ 运行 │ 45%     │ v2.1.3   │
│ orderer-02  │ ✅ 运行 │ 38%     │ v2.1.3   │
│ orderer-03  │ ⚠️ 告警 │ 92%     │ v2.1.3   │
└─────────────┴────────┴─────────┴──────────┘
💡 orderer-03 CPU 偏高，建议关注
```

**可查询的信息类型：**
- 服务/节点实时状态
- 运维知识库问答（操作手册、最佳实践、FAQ）
- 配置项及变更历史
- 资源容量水位（CPU/内存/磁盘/网络）
- 服务依赖拓扑

### 场景 2：告警整合

**用户在 Agent 页面看到推送：**
> 🔔 检测到告警，已为您整合分析

**Agent 后台流程：**
1. Agent 收到告警事件触发（或用户主动查询）
2. 把"告警整合"这一复合目标**拆解为原子问题**，通过 chrome-devtools-cli **串行**逐个提交：
   - "查询当前告警列表"
   - "查询 orderer-03 节点最近 30 分钟的事件"
   - "查询最近 1 小时的变更记录"
3. 分别轮询、提取多组回答（每轮 30s~3min）
4. **二次加工（核心价值）：**
   - 去重：相同根因的告警合并为一条
   - 关联：按时间线 + 拓扑关系串联合并
   - 分级：根据影响面和严重程度排序
   - 建议：匹配知识库给出处置建议

> ⏱ **耗时预期：** 3 条原子查询串行 ≈ 1.5~9 分钟。**不适合"秒级实时推送"**，适合"告警发生后几分钟内出整合简报"的准实时模式。

**Agent 页面展示：**
```
🚨 告警整合报告 - 09:52

【根因】orderer-03 节点 CPU 持续过高
├── 09:35  CPU > 80% 告警
├── 09:38  内存 > 90% 告警
├── 09:42  健康检查超时
└── 09:47  下游服务响应变慢

📊 影响范围：3 个下游服务
🔧 建议处置：检查 orderer-03 上的日志容器，疑似内存泄漏
📋 历史相似：5/12 曾出现同类问题（已通过重启临时解决）

[查看详情]  [一键确认]  [升级处理]
```

> ⚠️ **粒度说明：** 时间线（09:35/09:38…）为按 AI 返回的事件记录串联的示意；实际精度取决于 AI 助手返回的事件粒度，Agent 不补全/编造未返回的中间时刻。

**关键能力：** 用户侧 Agent 把零散的、多条的告警通过 AI 助手查询后聚合，形成一份有因果关系的报告，而非一堆独立告警。

### 场景 3：数据组合检索

**用户在 Agent 页面输入：**
> "最近一周哪些节点 CPU 超过 80%，同时内存超过 90%？列出趋势"

**Agent 后台流程：**
1. **AI 助手无法一次回答复合条件**，须拆解为原子问题、串行逐个提交：
   - Step 1（取实体）: "查询最近 7 天 CPU 利用率超过 80% 的虚拟机列表"
   - Step 2（取实体）: "查询最近 7 天内存利用率超过 90% 的虚拟机列表"
   - Step 3（下钻趋势，⚠️ **末尾必须加「用表格展示」**）: "查询这些节点最近一周的 CPU/内存利用率趋势，**用表格展示**"
2. 分别轮询、提取回答（趋势类用 `evaluate_script` 抓 `<table>` 原始数据，前提是 Step 3 带了「用表格展示」）
3. **二次加工：**
   - 取交集（同时满足 CPU + 内存条件的节点）
   - 合并趋势数据为统一图表
   - 标注异常时段和可能原因（**基于 AI 实际返回的粒度，不编造**）

> ⏱ **耗时预期：** 3 条串行 ≈ 1.5~9 分钟。首条数据检索可能返回"数据检索失败"，但系统会自动换数据集重试，须持续轮询直到「分析完成」。

**Agent 页面展示：**
```
📊 组合检索结果（最近 7 天）

筛选条件：CPU > 80% AND 内存 > 90%

命中节点：3 个
┌─────────────┬──────────────┬──────────────┐
│ 节点         │ CPU 周峰值   │ 内存周峰值   │
├─────────────┼──────────────┼──────────────┤
│ orderer-03  │ 95%          │ 94%          │
│ peer-07     │ 88%          │ 91%          │
│ kafka-02    │ 84%          │ 93%          │
└─────────────┴──────────────┴──────────────┘

📈 趋势：orderer-03 本周峰值逐日走高，建议扩容
```

> ⚠️ **粒度说明：** 上表展示 AI 助手实际返回的**周峰值**（max/min）。逐日/逐时段序列只有在 Step 3 带「用表格展示」、且 AI 确实返回了逐日数据时才可得；若 AI 仅返回周聚合值，Agent 据实展示周峰值，**不补全/编造逐日异常时段**（如"6/14 14:00~"这类精确时刻）。

**可扩展的组合查询：**
- 资源维度交叉：CPU × 内存 × 磁盘 × 网络
- 时间维度对比：本周 vs 上周 / 工作日 vs 周末
- 拓扑维度关联：指定节点 → 受影响服务 → 影响业务
- 变更维度关联：指定时间段变更 → 对应的异常指标

### 场景 4：运维周报

**用户在 Agent 页面操作：**
> 点击「生成本周运维周报」或设定定时任务每周一自动生成

**Agent 后台流程：**
1. 将"周报"拆解为 8 个**原子问题**，通过 chrome-devtools-cli **串行**逐个提交（⚠️ 趋势/聚合类问题带「用表格展示」）：

   | 序号 | 查询内容 |
   |------|----------|
   | Q1 | "当前环境总共有多少个节点？列出节点类型分布" |
   | Q2 | "本周新增和下线的节点有哪些？" |
   | Q3 | "本周所有节点的 CPU 平均利用率和峰值利用率，**用表格展示**" |
   | Q4 | "本周所有节点的内存平均利用率和峰值利用率，**用表格展示**" |
   | Q5 | "本周服务器的启停记录" |
   | Q6 | "本周告警总数、Top 5 告警来源、平均响应时间" |
   | Q7 | "本周执行了哪些变更操作？" |
   | Q8 | "本周服务的 SLA 达成情况" |

2. 逐个轮询、提取所有回答（数据检索失败时系统自动换数据集重试，须轮询至「分析完成」）
3. **二次加工：** 将分散的回答组装为结构化周报，补充趋势对比、风险提示

> ⏱ **耗时预期：** 8 条串行 ≈ **4~24 分钟**（单轮 30s~3min）。因此周报**必须异步生成、定时触发**（如每周一凌晨跑），不能做实时交互；生成完再推送至企微/邮件。

## AI 助手能力边界与约束（落地必读）

> 以下约束来自实际对接过程（见 `ops-data-multistep-query` 技能），是所有场景可行性的前提。**设计任何新场景前先对照本节。**

### 1. 只支持原子问题，复合问题必须拆解
AI 助手一次只接受一个原子问题。多条件（"CPU>80% 且 内存>90%"）、多目标（"TOP + 趋势"）必须由 Agent 拆成有序子问题串行提交，并在轮次之间回填变量（如上轮取到的实体名）。

### 2. 趋势/对比类问题必须追加「用表格展示」
此类问题默认渲染 ECharts 图表（canvas/SVG），DOM 无 `<table>`、无法抓取原始数据。问句末尾加「用表格展示」才会输出表格，Agent 才能 `evaluate_script` 抓到逐日/逐项原始数据。

### 3. 数据粒度如实呈现，不编造
AI 可能只返回**周/月聚合值**（max/min/avg）而非逐日序列。Agent 据实展示返回粒度，不补全、不编造未返回的中间数据点。

### 4. 单轮耗时 30s~3min，必须串行
一个对话框输入框同一时刻只能提交一个问题，多步查询天然串行。N 条查询总耗时 ≈ N × (30s~3min)。这直接决定：**实时/秒级响应不可行**，场景须设计为准实时（分钟级）或异步（定时）。

### 5. 「数据检索失败」不是终点
首轮检索常失败，但系统通常自动换数据集（如 `SYS.SERVER物理服务器性能分析数据集`）重试并在下方追加新分析块。须持续轮询直到「分析完成」或累计超时，**不要在首条失败就判定整体失败**。

### 6. SSL 多级自签名证书
云平台与鉴权域名存在多级自签名证书，跳转时可能反复触发证书错误。自启 MCP 须带 `--acceptInsecureCerts true`；复用 MCP 时用「点'高级' → '继续前往'」循环点击式绕过（最多 5 次）。

### 7. 登录态与会话
须检测登录页（含 textbox + 用户名 + 密码），冷启动需登录；会话超时需自动重新认证。多行输入框提交用「点击 → 输入 → 单独回车」，`submitKey` 在该控件上不可靠。

### 8. 绝不硬编码 selector / UID
页面 DOM 随版本变化。每一步操作前重新 `take_snapshot`，用语义文本（如"找'智能助手'文本"）动态定位元素，而非写死 CSS selector / UID。

### 9. 资源清理收尾
仅清理本次自启的浏览器/MCP；复用的已登录页面与共享 MCP 属于会话/用户，关闭会丢弃登录态、打断其他流程，**严禁清理**。

## 三、方案对比：Agent 中转 vs 用户直接访问

### 用户直接访问云平台 AI 助手

```
用户 → 浏览器登录云平台 → AI 助手对话框 → 逐条问答
```

### 用户侧 Agent 中转访问

```
用户 → Agent 页面 → Agent 后台(chrome-devtools-cli) → 云平台 AI 助手 → 二次加工 → 结构化展示
```

### 对比矩阵

| 维度 | 用户直接访问云平台 | 用户侧 Agent 中转 |
|------|-------------------|-------------------|
| **交互效率** | 每次手动登录、手动输入、单轮问答 | 自然语言一句话，Agent 自动拆解为多步查询 |
| **多步查询** | 用户需手动发起多轮，自己拼结果 | Agent 自动拆解、**串行**查询，自动合并（单对话框不支持并行） |
| **结果整合** | 原始文本回答，需用户自行提炼 | 二次加工：去重、关联、格式化为图表/卡片 |
| **告警处理** | 被动查看告警列表，逐条排查 | 自动聚合多源告警，根因分析 + 处置建议 |
| **报表生成** | 需手动逐项查询，手动拼凑周报 | 一键/定时生成，自动组装结构化报告 |
| **主动推送** | 需用户主动查看 | Agent 可定时巡检，异常主动推送 |
| **访问门槛** | 需要知道云平台地址、账号、操作路径 | 用户只需面对 Agent，降低使用门槛 |
| **上下文记忆** | AI 助手通常单会话，上下文有限 | Agent 可维持长期上下文，记住历史查询和偏好 |
| **权限隔离** | 用户直接持有云平台账号 | Agent 统一持证，可做权限收窄和审计 |
| **响应速度** | 直接访问，延迟最低 | 单轮 AI 数据检索 **30s~3min**、多步须串行；浏览器中转非主因，**AI 检索本身才是耗时大头** |
| **稳定性** | 直连，依赖少 | 依赖 chrome-devtools-cli + DOM 结构稳定，页面改版可能中断 |
| **开发维护** | 无额外开发 | 需维护浏览器自动化脚本、DOM 选择器、异常重试 |
| **功能边界** | 受限于 AI 助手原生能力 | 可跨多个系统组合（不止查一个云平台） |
| **数据安全** | 用户直接访问，行为可审计 | Agent 代理访问，需额外审计 Agent 侧行为 |

### 综合评价

| 方面 | 结论 |
|------|------|
| **性能** | 直连更快；Agent 中转的单轮 AI 检索 30s~3min、且须串行，是主要性能瓶颈 |
| **功能** | Agent 中转显著更强（多步查询、结果整合、跨系统组合） |
| **易用性** | Agent 中转显著更好（统一入口、自然语言、结构化展示） |
| **可靠性** | 直连更稳定；Agent 中转依赖 DOM 结构，维护成本高 |
| **安全性** | 各有侧重；Agent 中转多一层代理，需额外审计 |

## 四、使用场景（优势导向）

> 以下场景充分发挥 Agent 中转方案的优势，是直连方案难以覆盖的高价值场景。

### 场景 A：运维一站式工作台

**目标用户：** 运维工程师、SRE

**描述：**
Agent 页面作为运维统一入口，用户不需要登录多个系统。一句话即可触发跨平台、跨维度的综合查询。

**典型交互：**
- "帮我看看现在整个生产环境有没有异常"
- Agent 自动向 AI 助手**串行**发起 5~8 条原子查询（告警、CPU、内存、磁盘、SLA、变更），合并为一份健康报告

> ⏱ **耗时预期：** 5~8 条串行 ≈ **2.5~24 分钟**。因此是**准实时**（分钟级）健康报告，不是秒级仪表盘。若需秒级刷新，应叠加独立监控数据源，AI 助手查询用于"综合研判"而非"实时刷新"。

**核心优势：** 多步查询 + 结果整合，直连方案做不到。

---

### 场景 B：定时巡检与主动告警

**目标用户：** 运维负责人、值班人员

**描述：**
Agent 设定定时任务（如每 30 分钟），自动通过 AI 助手巡检关键指标。发现异常时主动推送到用户，并附上整合分析。

**典型交互：**
- Agent 后台按周期自动查询（建议巡检查询数 ≤ 4~5 条，以控制单轮总耗时）
- 发现 orderer-03 CPU 异常 → 主动推送："orderer-03 CPU 持续升高，已关联最近变更，疑似 6/12 内存调整后的连锁反应"
- 用户可直接在 Agent 页面确认或升级处理

> ⏱ **可行性约束：** 单轮 30s~3min 且串行。若巡检 5 条查询，单次约 2.5~15 分钟——**"每 30 分钟巡检"是可行的下限周期**；更短周期（如每 5 分钟）应改用独立监控指标直接采，AI 助手仅做周期性深度研判。

**核心优势：** 主动推送 + 自动关联分析，用户不需要一直盯着云平台。

---

### 场景 C：自动化周报/月报

**目标用户：** 运维管理者、项目负责人

**描述：**
设定定时任务，Agent 每周一自动向 AI 助手发起 8~12 条**串行**查询，组装为完整周报，直接发送到指定渠道（企微/邮件/钉钉）。

> ⏱ **耗时预期：** 8~12 条串行 ≈ **4~36 分钟**。异步生成、定时触发（如每周一凌晨），生成完再推送——**这是与 AI 助手能力最匹配的场景**（不要求实时）。

**核心优势：**
- 直连方案下用户需要花 1~2 小时手动查询拼凑
- Agent 方案零人工，且格式统一、可追溯

---

### 场景 D：故障应急辅助

**目标用户：** 值班工程师、一线运维

**描述：**
告警触发时，Agent 立即向 AI 助手发起关联查询（拓扑、历史、变更），推送一份包含根因推测和处置建议的简报。

**典型交互：**
- 收到 P1 告警 → Agent 自动**串行**查询（4 条）：
  - "orderer-03 当前告警详情"
  - "orderer-03 依赖的下游服务"
  - "最近 2 小时的变更记录"
  - "历史上 orderer-03 CPU 高的处置方案"
- 生成应急简报并推送

> ⏱ **可行性约束（重要修正）：** 4 条串行 ≈ **2~12 分钟**，**不可能"10 秒内"出简报**。合理的应急设计是**两段式**：
> - **秒级**：用预置预案库 / 独立监控数据，秒级推送"告警卡片 + 指南链接"应急先手；
> - **分钟级**：AI 助手 4 条关联查询跑完（2~12 分钟）后，再追加"根因推测 + 处置建议"深度简报。
>
> 即"秒级触发、分钟级研判"，而非"10 秒出全量简报"。

**核心优势：** 自动化关联查询 + 整合可节省 5~15 分钟人工排查时间（前提是接受分钟级延迟，而非秒级）。

---

### 场景 E：运维知识自助服务

**目标用户：** 开发团队、非运维人员

**描述：**
开发人员遇到运维相关问题（部署流程、配置说明、排障方法），直接问 Agent，Agent 转发给 AI 助手获取知识库答案，补充实际环境数据后返回。

**典型交互：**
- 开发："支付服务部署在哪几个节点？当前版本是什么？"
- Agent 查询后回答，并补充："支付服务当前部署在 peer-03/04/05，版本 v3.2.1，6/14 刚更新过"

> ⏱ **耗时预期：** 知识库问答通常比数据检索快，单轮约 10s~1min；环境数据补充类查询仍按 30s~3min 预估。整体可做到"问一句、1 分钟内回"，是体验最接近实时的场景。

**核心优势：** 降低非运维人员使用门槛，不需要知道云平台怎么操作。

---

### 场景 F：多云/多平台联合查询

**目标用户：** 架构师、技术负责人

**描述：**
Agent 不仅对接一个云平台 AI 助手，还可以分别驱动多个平台页面，将跨平台的运维数据组合查询（仍为串行执行，耗时按平台叠加）。

**典型交互：**
- "对比一下华为云和阿里云上的节点资源利用率"
- Agent 分别打开两个平台的 AI 助手，查询后合并对比

> ⏱ **可行性约束：** 多平台查询**串行耗时叠加**（平台 A 的 N 条 + 平台 B 的 M 条），且每个平台独立处理 SSL/登录/切换。建议每平台查询数精简，整体按"分钟级~十分钟级"预期；适合规划性对比，不适合实时联动。

**核心优势：** 直连方案下用户需要分别登录两个平台；Agent 统一中转。

## 五、技术实现要点

### chrome-devtools-cli 交互关键点

> ⚠️ 与初版设想的关键差异：**串行而非并行**、**动态 snapshot 定位而非固定 selector**、**长超时轮询（非 30s）**、**提交用单独回车**。

```typescript
// 伪代码示例（贴合 ops-data-multistep-query 实际过程）
// 原则：每步 take_snapshot 动态定位，绝不硬编码 selector/UID；查询串行执行

// 单轮原子查询
async function askAtomic(question: string): Promise<string> {
  // 1. 动态定位输入框（每轮重新点击获取焦点）
  const snap = await chromeDevtools.takeSnapshot();
  const inputUid = findBySemantic(snap, '您可以用"产品+问题"的形式描述问题');
  await chromeDevtools.click(inputUid);

  // 2. 输入 + 单独回车提交（多行框 submitKey 不可靠）
  await chromeDevtools.type_text(question);
  await chromeDevtools.takeSnapshot();          // 确认 value 已写入
  await chromeDevtools.press_key('Enter');

  // 3. 长超时轮询阶段状态（单轮 30s~3min）
  await pollUntilDone({
    maxWaitSeconds: 180,
    pollIntervalSeconds: 15,
    isDone: s => hasText(s, '分析完成'),
    isFailed: s => hasText(s, '抱歉，助手暂时无法回答您的问题'), // 失败重试 1 次
    // 注意："数据检索失败" 不算终点——系统会自动换数据集重试，继续轮询
  });

  // 4. 提取：趋势类问题带了"用表格展示"时，用 evaluate_script 抓表格更稳
  if (question.includes('用表格展示')) {
    return await chromeDevtools.evaluate_script(
      "() => Array.from(document.querySelectorAll('table tbody tr')).map(r => r.innerText)"
    );
  }
  return extractAnswerText(await chromeDevtools.takeSnapshot());
}

// 多步查询：串行（单对话框不支持并行），轮次间回填变量
async function combinedQuery(plan: QuestionPlan[]): Promise<MergedResult> {
  const results = [];
  for (const step of plan) {
    const q = step.render(results);   // 可引用上一步结果中的变量（如实体名）
    const ans = await askAtomic(q);   // 串行 await，不要 Promise.all
    results.push(ans);
  }
  return mergeAndProcess(results);     // 去重/关联/如实粒度
}

// 前置：首次需处理 SSL（--acceptInsecureCerts 或循环点击绕过）、登录、切换云平台
// 收尾：仅清理本次自启的浏览器/MCP，复用的登录态与会话严禁回收
```

### 关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 云平台页面改版导致 DOM 变化 | 自动化中断 | **每步 take_snapshot 动态语义定位**，绝不硬编码 selector/UID；建立监控告警 |
| AI 助手回答不稳定/格式不一致 | 提取结果解析失败 | 容错解析 + 趋势类问题强制「用表格展示」并 `evaluate_script` 抓表格 |
| **单轮耗时 30s~3min + 串行** | 用户感知"慢"、实时场景不可行 | 异步流式展示（先"查询中"逐步更新）；场景设计为准实时/异步，不承诺秒级 |
| **「数据检索失败」误判为终点** | 整体查询被错误中止 | 系统会自动换数据集重试，**持续轮询**至「分析完成」或累计超时 |
| **AI 仅返回聚合粒度（周 max/min）** | 逐日趋势缺失 | 如实展示返回粒度，不编造逐日序列；需逐日数据时单独追加「用表格展示」查询 |
| 会话超时/登录态丢失 | 查询失败 | 自动检测登录页，支持自动重新认证；处理 SSL 多级自签名证书 |
| **多步查询误用并行** | 单输入框冲突、结果错乱 | **强制串行**（for...of + await），轮次间回填变量 |
| **清理时误关复用页面/共享 MCP** | 丢弃登录态、打断其他流程 | 仅清理本次自启资源；按"起始状态判定表"决定 close_page / stop |

## 六、总结

Agent 中转方案的核心价值在于 **"多步查询 → 自动整合 → 结构化展示"**，把用户从"在云平台手动逐条问答"升级为"跟 Agent 说一句话，拿结构化结果"。

但能力边界必须前置：**AI 助手只接受原子问题、单轮 30s~3min、必须串行、趋势类须带「用表格展示」、数据粒度如实呈现**。因此——

- **简单单条信息查询**：直连云平台更直接高效；
- **告警整合、组合检索、报表生成、主动巡检**等复合场景：Agent 中转优势明显，但**均为分钟级~十分钟级，不是秒级**；
- 任何宣称"实时/秒级"的场景（尤其应急），须用预案库 + 独立监控数据补足秒级先手，AI 助手查询只做分钟级深度研判。

**落地建议：**
- **首推场景 C（周报自动化）**——异步、定时、不要求实时，与 AI 助手能力最匹配，价值最高；
- **次推场景 D（故障应急辅助）**——价值高，但须按"秒级触发 + 分钟级研判"两段式实现，不能承诺 10 秒出全量简报；
- 场景 A/B（一站式、巡检）按准实时定位，注意控制单次查询条数与巡检周期下限。
